## Приложение с использованием сервисов (Android Services)

### Описание проекта

Данный проект представляет собой учебное Android-приложение, разработанное на Kotlin с использованием Jetpack Compose, демонстрирующее работу Android Service и механизмов фонового выполнения задач.

Проект реализует несколько вариантов работы сервисов:

* простой Service;
* stopSelf(startId);
* START_STICKY / START_NOT_STICKY / START_REDELIVER_INTENT;
* взаимодействие через PendingIntent;
* выполнение задач в отдельных потоках;
* передачу результатов из сервиса в UI.

Приложение создано в рамках практической работы №18 по дисциплине:
«Разработка мобильных приложений».

---

# Цель работы

Изучить:

* жизненный цикл Android Service;
* запуск и остановку сервисов;

# Основные экраны приложения

## 1. Simple Service

Экран демонстрирует:

* запуск сервиса;
* остановку сервиса;
* работу сервиса в фоне;
* выполнение задачи в отдельном потоке.

### Как работает

При нажатии:

* Start service → вызывается startService();
* Stop service → вызывается stopService().

Сервис:

1. создаётся через onCreate();
2. запускает поток;
3. выводит значения в лог;
4. завершает работу через stopSelf().

### Что показать на защите

Показать:

* запуск сервиса;
* работу ProgressIndicator;
* вывод логов;
* остановку сервиса;
* что UI не блокируется.

### Главное объяснение

По умолчанию Service работает в main thread, поэтому тяжёлые операции нужно выносить в отдельный поток.

---

# 2. stopSelf(startId)

Экран демонстрирует:

* работу startId;
* stopSelf(startId);
* stopSelfResult(startId);
* последовательное и параллельное выполнение задач.

### Как работает

При запуске сервис получает:

* startId;
* время выполнения задачи.

Создаётся Runnable-задача и передаётся в ExecutorService.

Есть два режима:

* последовательный (1 поток);
* параллельный (3 потока).

После завершения задача вызывает:

```kotlin
stopSelf(startId)
```

или:

```kotlin
stopSelfResult(startId)
```

### Что показать на защите

## Последовательный режим

Показать:

* задачи выполняются по очереди;
* сервис завершается после последней задачи.

## Параллельный режим

Показать:

* задачи выполняются одновременно;
* stopSelf(startId) может остановить сервис раньше завершения других потоков.

### Главное объяснение

startId – это номер запуска сервиса.

Android останавливает сервис, когда stopSelf() вызывается для последнего startId.

---

# 3. START_STICKY / START_NOT_STICKY / START_REDELIVER_INTENT

Экран демонстрирует:

* поведение сервиса после уничтожения процесса системой.

### Режимы

| Режим                  | Поведение                                 |
| ---------------------- | ----------------------------------------- |
| START_NOT_STICKY       | сервис не перезапускается                 |
| START_STICKY           | сервис перезапускается без intent         |
| START_REDELIVER_INTENT | сервис перезапускается с последним intent |

---

## Как работает

Пользователь выбирает режим.

Сервис сохраняет:

* режим запуска;
* payload.

При имитации kill process:

* сервис останавливается;
* приложение имитирует восстановление сервиса.

---

## Что показать на защите

### START_NOT_STICKY

Показать:

* сервис не восстанавливается.

### START_STICKY

Показать:

* сервис перезапускается;
* intent может быть null.

### START_REDELIVER_INTENT

Показать:

* сервис перезапускается;
* intent восстанавливается.

### Главное объяснение

Эти режимы определяют поведение Android после уничтожения процесса приложения.

# 4. PendingIntent Service Result

Экран демонстрирует:

* передачу результатов из сервиса;
* PendingIntent;
* BroadcastReceiver;
* обновление UI после выполнения фоновой задачи.

---

## Как работает

1. UI запускает сервис.
2. В Intent передаётся:

   * taskCode;
   * время выполнения;
   * PendingIntent.
3. Сервис выполняет задачу в потоке.
4. После завершения отправляет результат через:

```kotlin
PendingIntent.send()
```

5. BroadcastReceiver получает результат.
6. Repository обновляет состояние.
7. Compose автоматически обновляет UI.

---

## Что показать на защите

Показать:

* запуск задач;
* изменение статусов:

  * Waiting
  * Started
  * Finished
* обновление результата без перезапуска экрана.

# Почему используются потоки

Service не создаёт отдельный поток автоматически.

Если выполнять тяжёлую работу внутри onStartCommand(), интерфейс начнёт зависать.

Поэтому используются:

* thread {};
* ExecutorService.
